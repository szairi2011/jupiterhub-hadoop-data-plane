Isolation des ressources et gouvernance apportées par JupyterHub + Livy + Spark Resource isolation and control gained by adopting JupyterHub + Livy + Spark

❌ Avant — Machine unique (e.g. Jump Box) ❌ Before — Single Machine (e.g. Jump Box)

✗ Jupyter, Livy et Spark sur le même hôte Jupyter, Livy and Spark running on the same host
✗ Aucune isolation utilisateur — espace de processus partagé No user isolation — shared process space
✗ Aucune limite de ressources — un job peut affamer les autres No resource limits — any job can starve others
✗ Impossible de monter en charge au-delà d’une machine Cannot scale beyond one machine
✗ Aucun contrôle sur les sessions concurrentes No control over concurrent sessions
✗ Toutes les ressources du cluster accessibles à un seul utilisateur Full cluster resources available to a single user

✅ Après — Plateforme distribuée ✅ After — Distributed Platform

✓ Chaque utilisateur dispose d’un pod dédié et isolé Each user gets a dedicated, fully isolated pod
✓ Quotas CPU et RAM appliqués par pod via la configuration CPU and RAM quotas enforced per pod by configuration
✓ Pods Notebook distribués et scalables sur les nœuds K8s Notebook pods distributed and scaled across K8s nodes
✓ Nombre max de sessions Livy simultanées limité par configuration Max concurrent Livy sessions capped by configuration
✓ Mémoire des workers et executors Spark configurée centralement Spark worker and executor memory centrally configured
✓ Files YARN pour la priorisation des jobs (администрé côté cluster) YARN queues for priority tiers (managed by cluster admin)
